## Overall Impression

This went from a faithful-but-flawed translation (28) to a genuinely shippable mobile product (32). The P0 (one-tap-per-egg) and P1 (contrast) are resolved cleanly; the field worker can now register real volumes and read every label in sunlight. Biggest single opportunity: replace the native `window.confirm` with an in-language confirm so the one destructive action doesn't visually fall out of the app.

## What's Working

- **Status visibility is now complete.** Skeleton shimmer (with reduced-motion fallback) on Hoy/Inventario, toast on save, sync banner, online/offline pill, aria-live regions. A user always knows what the system is doing.
- **Draft survives navigation.** Lifting the Registrar draft into AppShell + `hidden` instead of unmount means an interrupted count isn't lost on tab switch — the exact failure that broke Casey before.
- **Honest data discipline held.** Real API where it exists; explicit "Próximamente" / empty-card where it doesn't. Contrast fixes verified against the project's own stated WCAG AA bar.

## Priority Issues

- **[P2] `window.confirm` for logout breaks the design language.** It's the one raw OS dialog in an otherwise cohesive flat-mint app — unstyleable, off-brand, and visually jarring on mobile.
  - **Why it matters:** Consistency heuristic; the single destructive action looks like it belongs to a different product.
  - **Fix:** Inline two-step confirm on the logout button (tap → "Confirmar salida?" with cancel), or a small styled bottom-sheet. Keep it in-language and in-palette.
  - **Suggested command:** `/impeccable harden`

- **[P2] Ajustes bottom-nav icon is an Egg, not a settings glyph.** Every other nav item's icon matches its meaning; Ajustes reuses the brand egg, which reads as a second "huevos" destination.
  - **Why it matters:** Recognition/consistency — a worker scanning the bar can't map the egg to "settings."
  - **Fix:** Swap to lucide `Settings` (gear). One-line change.
  - **Suggested command:** `/impeccable polish`

- **[P2] Hoy production meta label is ambiguous.** The stat shows production total with a sub-label that flips between "X rotos" and "huevos buenos" depending on whether rotos exist — the unit of the big number itself is never stated.
  - **Why it matters:** A glanceable dashboard number should carry its own unit; mixing the rotos count into the meta line muddies it.
  - **Fix:** Label the value "huevos" consistently and show rotos as a separate small stat or chip, not as the meta of production.
  - **Suggested command:** `/impeccable clarify`

- **[P3] Numeric inputs have no upper bound / paste guard.** Collection and sale quantity accept arbitrarily large pasted values; only a floor (0) and integer-floor are enforced.
  - **Why it matters:** A fat-fingered paste registers 9,999,999 eggs with no warning; sale over-quantity is caught by the backend but collection isn't.
  - **Fix:** Add a sane `max` (e.g. per-category daily ceiling) or a confirm above a threshold.
  - **Suggested command:** `/impeccable harden`

- Mid-form refresh loses unsaved draft (state is in-memory, not persisted) — acceptable for v1 but worth a note.
- `--z-*` tokens exist in `:root` but the new toast/nav/skeleton use literal z-index (40/30); fold them into the semantic scale.

## Questions to Consider

- Should logout confirmation become a reusable styled confirm primitive, given "precios/usuarios/cerrar sesión" will all eventually need destructive/confirm patterns?
- Is there a realistic per-category daily ceiling that would make a numeric `max` meaningful, or is a soft "that seems high — confirm?" threshold the better guard?
- Does the in-memory draft need to survive a hard refresh (persist to IndexedDB), or is tab-switch persistence enough for the field flow?
